# 自研还是外购：为什么 Core 可能要自己做 PDF 渲染

> 本文档讨论 FeatherDoc Core 在 PDF 渲染问题上"自研 vs 外购"的决策依据。
>
> 阅读前置：[PDF 渲染策略](pdf-rendering-strategy.md)。

## 简短结论

| 选项 | 何时合适 | 何时不合适 |
|---|---|---|
| **进程调外部 Office**（LibreOffice / Word） | 当下、绝大多数场景 | 性能瓶颈、依赖膨胀不可接受时 |
| **商业库 Aspose.Words** | 有付费预算、需要单二进制部署 | 商业项目想避开年费 |
| **自研 Core 渲染** | 长期、有团队、有特定子集场景 | 团队 ≤ 2 人、商业未稳定 |
| 重写 LibreOffice | **永远不合适** | 任何时候 |

本文档解释为什么。

## "Word → PDF" 不是一个原子能力

市面上所有号称能做 Word↔PDF 的方案（Aspose.Words / Spire.Doc / LibreOffice / docx4j+iText / OnlyOffice / Pandoc+LaTeX），内部都是三段式拼装：

```text
Word → PDF 库  =  docx 解析  +  排版引擎  +  PDF 写出器
PDF → Word 库  =  PDF 解析   +  版面还原   +  docx 写出器
```

**没有任何一个库能跳过排版引擎直接"转换"**。所谓"转换"这个词本身就是误导——Word 和 PDF 在数据模型上根本不对等：

- **docx** 描述的是**逻辑结构**：这是一个段落、这是一个表格、这段文字用这个样式。它**不知道**最终会排成几页、每行断在哪。
- **PDF** 描述的是**绘制指令**：在坐标 (x, y) 用字体 F 画字形 G。它**不知道**这些字形原本属于哪个段落、哪个表格。

从 docx 到 PDF 要补上"排版"这一步（把逻辑结构变成绘制指令），从 PDF 到 docx 要补上"版面逆向还原"（把绘制指令猜回逻辑结构）。

## 两个方向难度完全不对称

| 方向 | 本质问题 | 工程难度 | 有没有"正确答案" |
|---|---|---|---|
| **Word → PDF** | 排版（断行/分页/表格/浮动/页眉页脚） | 大 | ✅ 有，按 OOXML 规范排即可 |
| **PDF → Word** | 版面逆向还原（段落识别、表格识别、阅读顺序） | 更大 | ❌ 没有，只能靠启发式 + ML 猜 |

PDF → Word 在学术上属于 **document layout analysis**。商业产品（ABBYY / Adobe / Solid Documents）都做不到 100% 还原，开源方案（pdf2docx / Marker / Nougat）效果更有限。

因此，Core 不应把 PDF → Word 当成 PDF 渲染路线的一部分。如果未来做反向转换，也应作为独立的
PDFium → AST 适配层研究，并且按 best-effort 能力对外表达。

## 自研 vs 外购的真实对比

### 路径 1：进程调外部 Office

**做法**：Core CLI 内部 `system("soffice --headless --convert-to pdf")`，或下游通过进程调用。

| 维度 | 评分 |
|---|---|
| 工作量 | 0（已经能跑） |
| Core 二进制大小 | 不变 |
| 用户机器 | 需要装 LibreOffice/Word |
| 性能 | 启动 5–10s，单文档 OK，批量慢 |
| 保真度 | 高（就是 LO/Word 自己） |
| 长期维护 | 跟随 LO 上游升级即可 |

**适用**：当下绝大多数场景。

### 路径 2：商业库 Aspose.Words

**做法**：Core 链接 / 调用 Aspose.Words for C++（商业付费）。

| 维度 | 评分 |
|---|---|
| 工作量 | 1–2 周接入 |
| Core 二进制大小 | +50 MB |
| 用户机器 | 无外部依赖 |
| 性能 | 快（单二进制） |
| 保真度 | 高 |
| 成本 | $1000+/年（看授权） |
| 长期维护 | Aspose 升级跟进 |

**适用**：有付费预算、需要单二进制部署的商业产品。

### 路径 3：自研 Core 渲染（本研究方向）

**做法**：Core 自己写排版引擎 + PDFio PDF 输出，未来按需要评估 Skia 等复杂绘制后端。

| 维度 | 评分 |
|---|---|
| 工作量 | 9–14 个月（合同子集，1 人专职） |
| Core 二进制大小 | +30–80 MB（按可选模块） |
| 用户机器 | 无外部依赖 |
| 性能 | 极快（无外部启动） |
| 保真度 | Phase 1 中等，长期可逼近 LO |
| 持续维护 | 永远（OOXML 标准持续演进） |
| 商业差异化 | ✅ 有 |

**适用**：长期、有团队、特定业务场景需要自主可控。

### 路径 4：重写 LibreOffice

**永远不要做。** 详见下文 §"为什么不能直接抽离 LibreOffice"。

## 为什么不能直接抽离 LibreOffice

不少人会想：LibreOffice 已经开源，能不能"抽出排版引擎"塞进 Core？

**两个独立问题同时卡死这个方案**：

### 问题 1：技术上抽不出来

| 模块 | 大小 | 能不能去掉 |
|---|---|---|
| GUI / VCL | ~80 MB | ✅ headless 不需要 |
| Calc / Impress / Draw | ~150 MB | ✅ 你只要 Writer |
| 多语言资源 | ~50 MB | ✅ 可裁 |
| **Writer 排版引擎** | ~30 MB | ❌ **核心** |
| **OOXML 解析** | ~10 MB | ❌ 必须 |
| **HarfBuzz + FreeType + ICU** | ~50 MB | ❌ 必须 |
| **PDF 输出** | ~10 MB | ❌ 必须 |
| 服务 / 配置 / 构建框架 | ~30 MB | ❌ 拆了跑不起来 |

理论最小可用集**至少 130 MB+**。LibreOffice 团队 22 年都没把这个轻量子集做出来，模块耦合太深。

### 问题 2：许可证不会因为换包装就变化

LibreOffice 代码大部分是 **MPL 2.0 / LGPL 3.0+**。这两个都是 **copyleft**：

- 复制 MPL 文件到 Core，那些文件**仍然是 MPL**，不会变 Apache
- 修改 MPL 文件**必须开源修改部分**
- LGPL 静态链接需要提供 .o 文件让用户能重链接

**结论**：把 LibreOffice 代码塞进 Core 既技术上做不到，许可证上也得不到任何便利。

### 业界没人这么做

| 产品 | 做法 |
|---|---|
| Notion / Confluence 导出 | 服务端跑 LO（进程） |
| OnlyOffice | 自研 + AGPL 开源 / 商业双授权 |
| WPS | 自研（30+ 年积累） |
| Google Docs | 自研（20+ 年） |
| Adobe / Aspose | 自研商业产品 |

**没有任何成功产品是"抽离 LibreOffice 集成进自家库"** —— 既不合法可行，也不工程可行。

## 自研的真实价值

那 Core 为什么还要考虑自研？

### 价值 1：性能

- LibreOffice 启动 5–10 秒
- Core 自研可以做到启动 < 100ms
- 批量场景（1000 份合同）：LO 几小时，Core 自研可能几分钟

### 价值 2：依赖独立

- 用户机器不需要装 Office
- 部署到无 GUI 服务器更方便
- 容器化更小

### 价值 3：商业差异化

- 下游商业产品可以宣传"自研引擎，无外部依赖"
- 对 Word 兼容性、对中文场景的优化可以**自主控制**
- 长期可演进为商业引擎（类似 Aspose 路径）

### 价值 4：技术品牌

- 一个真正能做出 OOXML 渲染的开源 C++ 项目，社区影响力远超"包了一层 LO"
- 吸引高质量贡献者
- 学术 / 工业研究合作机会

## 自研的真实代价

### 代价 1：工程量

- Phase 1（合同子集）：9–14 个月，1 人专职
- Phase 1–4 全套：3–5 年
- 与 Word 视觉一致性：永远在调

### 代价 2：维护

- OOXML 标准持续演进
- 各种边角文档兼容问题修不完
- 字体 / 国际化的长期负担

### 代价 3：心智负担

- 团队会持续被"为什么 Word 显示是这样的"问题困扰
- 用户报 bug 比 LO 时代复杂

## 决策框架

启动自研前，回答以下问题。**全部"是"才启动**：

- [ ] Core 已经稳定，有持续下游商业用户？
- [ ] 团队有 1+ 人能专职 9–14 个月？
- [ ] 已经验证外部 Office（路径 1）确实是瓶颈，不是想象的瓶颈？
- [ ] 商业 Aspose（路径 2）成本评估完毕，确认走自研更划算？
- [ ] 准备好持续 3–5 年的工程投入？

## 一句话总结

> **不要把"做一个 Word 转 PDF 库"当成项目目标**，那只是换了个说法描述"写排版引擎 + 接 PDF 后端"。
>
> Core 自研 PDF 渲染是一条**长期路径**，不是当下的工程任务。
> 当下能跑、能赚钱的方案是路径 1（外部进程）和路径 2（商业库）。
> 路径 3（自研）是 Core 的长期愿景，需要稳定团队和持续投入。
>
> 如果扩展为 Word ⇄ PDF 双向转换器，反向路线应明确拆成"PDFium 解析 + AST 重建 + docs 库写出"。
