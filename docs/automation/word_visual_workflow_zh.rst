自动化 Word 可视检查与修复流程
================================

本说明把 FeatherDoc 的 Word 产物验证拆成两段独立流程：

- **视觉检查阶段**：只收集证据、给结论，不修改文档。
- **修复闭环阶段**：仅在用户明确要求修复，或当前任务本身就是“检查并修复”时进入。

核心原则是把 ``Microsoft Word`` 里的**最终肉眼可见效果**作为验收标准，而不是只看单元测试、XML 结构或导出的中间数据。


目录约定
--------

推荐把所有人工或自动化可视检查产物都集中在 ``output/word-visual-smoke/`` 下：

.. code-block:: text

    output/word-visual-smoke/
    ├── <当前待检文档>.docx
    ├── <当前待检文档>.pdf
    ├── evidence/
    │   ├── contact_sheet.png
    │   └── pages/
    ├── report/
    │   ├── summary.json
    │   ├── review_checklist.md
    │   └── <可选> final_review.md
    └── repair/
        ├── fix-01/
        │   ├── candidate.docx
        │   ├── evidence/
        │   └── report/
        └── fix-02/

约定含义如下：

- ``evidence/`` 保存截图、分页渲染、联系图等可视证据。
- ``report/`` 保存结构化摘要、人工审查结论和修复建议。
- ``repair/`` 保存每一轮修复候选文档，避免直接覆盖原始输入。

仓库中还提供两份可复用模板：

- ``docs/automation/final_review_template.md``：人工可读的最终结论模板。
- ``docs/automation/review_result_template.json``：机器可读的审查结果模板。
- ``docs/automation/task_prompt_review_only_template.md``：只检查、不修复的任务提示词模板。
- ``docs/automation/task_prompt_review_and_repair_template.md``：检查失败后继续修复的任务提示词模板。


如何驱动 AI 执行
----------------

推荐不要每次手工拼提示词，而是通过 ``scripts/prepare_word_review_task.ps1``
先生成一份标准任务包。这样 AI 的输入边界、输出目录和修复目录都会固定下来。

只检查模式：

.. code-block:: powershell

    powershell -ExecutionPolicy Bypass -File .\scripts\prepare_word_review_task.ps1 `
        -DocxPath C:\path\to\target.docx `
        -Mode review-only

检查并修复模式：

.. code-block:: powershell

    powershell -ExecutionPolicy Bypass -File .\scripts\prepare_word_review_task.ps1 `
        -DocxPath C:\path\to\target.docx `
        -Mode review-and-repair

脚本会创建一份任务目录，并输出：

- ``task_prompt.md``：直接发给 AI 的完整任务指令。
- ``task_manifest.json``：机器可读的任务元数据。
- ``evidence/``：截图和分页证据目录。
- ``report/``：本轮检查报告目录。
- ``repair/``：修复闭环目录。

推荐使用方式：

1. 先运行 ``prepare_word_review_task.ps1`` 生成任务包。
2. 打开生成的 ``task_prompt.md``，将其完整内容发给 AI。
3. 让 AI 把检查结论和截图依据回写到任务包里的 ``report/`` 目录。
4. 若任务模式是 ``review-and-repair``，则允许 AI 在 ``repair/`` 下开展多轮修复与回归。

这样后续你不需要反复描述“文档在哪里、输出写哪里、是否允许修复、报告写什么格式”，脚本已经替你固定好了。


最简操作
--------

如果你只想用最短路径把这条链路跑起来，按下面四步即可：

1. 跑 ``prepare_word_review_task.ps1``。
2. 打开生成的 ``task_prompt.md``，把整段内容发给 AI。
3. 明确要求 AI 把结论和截图依据回写到 ``report/``。
4. 如果模式是 ``review-and-repair``，允许它在 ``repair/`` 下做多轮修复和完整回归。


视觉检查阶段
------------

视觉检查阶段只负责回答一件事：**当前文档是否通过版式检查**。

进入该阶段时，执行代理必须遵守以下门禁：

- 先确认目标文档绝对路径存在，且扩展名为 ``.doc`` 或 ``.docx``。
- 必须明确记录将要打开的绝对路径，不能只说“打开文档”。
- 不能只凭 ``WINWORD.EXE`` 存在就视为成功，必须验证目标文档已经在 Word 中打开。
- 必须假设用户会同时操作电脑并抢焦点，所以每次截图前都要重新确认当前前台窗口仍然是 Word。
- 必须再次验证当前活动文档是目标文档，而不是空白页、开始页、最近文件页，或另一个同名之外的文档。

推荐固定执行顺序：

1. 验证目标文档路径、文件名、修改时间。
2. 启动 Word，并显式打开目标文档。
3. 校验当前窗口是否为 Word，当前活动文档是否匹配目标文档。
4. 切到适合人工排版审查的视图，例如打印布局，并调整到能看清页面主体的缩放比例。
5. 对每一页或关键页面执行固定动作：

   - 重新验证焦点窗口是 Word。
   - 重新验证活动文档仍是目标文档。
   - 使用 ``Screenshot`` 捕获页面视觉内容。
   - 若有可疑点，再用 ``Snapshot`` 校验焦点、窗口标题、控件状态和遮挡情况。

6. 按截图内容检查以下项目：

   - 一级标题、二级标题的位置和层级感是否正确。
   - 正文是否存在异常空行、错位、截断、重叠、丢字。
   - 表格边框、列宽、行高、跨页、标题行重复、分页断裂是否异常。
   - 表格单元格纵排/旋转文字是否可读，是否出现裁切、边框漂移、与相邻单元格错位。
   - 窄单元格中的 RTL/LTR/CJK 混排是否出现顺序错乱、异常换行、标点翻转、字体漂移或重叠。
   - 图片、图注、文字环绕、锚点位置是否异常。
   - 页码、页眉、页脚是否存在，位置是否正常。
   - 是否出现元素重叠、超出页面、分页突变、大片空白、编号错乱。

若以下任一条件成立，则**不能**输出“通过”：

- 截图前没有重新确认焦点窗口；
- 无法确认当前文档就是目标文档；
- 截图被用户抢焦点、弹窗或其他窗口污染；
- 只看了结构或 XML，缺少最终视觉证据。

视觉检查阶段的默认输出分级建议：

- ``通过``：证据完整，且未发现明显版式问题。
- ``不通过``：发现至少一个可复现、可截图定位的版式问题。
- ``无法判定``：证据不足，或无法确认截图对应目标文档。


修复闭环阶段
------------

修复闭环阶段默认**不自动进入**。只有在以下场景才进入：

- 用户明确要求“发现问题后继续修复”；
- 当前任务本身就是“检查并修复文档生成问题”；
- 前一轮视觉检查已经输出 ``不通过``，且用户没有禁止修改生成逻辑。

修复闭环必须遵守以下约束：

- 不要直接覆盖原始输入文档。
- 每一轮修复都在 ``repair/fix-XX/`` 目录下产出候选文档和证据。
- 每修完一轮，都必须重新执行完整视觉检查，而不是只盯住单个怀疑点。
- 只有在新一轮证据显示“通过”后，才能把修复结果标记为候选成功版本。

推荐修复流程如下：

1. 读取最近一轮视觉检查报告，提取问题页码、元素类型和截图依据。
2. 先判断问题属于哪一层：

   - 标题、正文、段落间距、缩进异常：优先检查段落和样式写入逻辑。
   - 表格宽度、边框、跨页、合并、单元格纵排/旋转文本、窄格混排异常：优先检查 ``Table`` / ``TableRow`` / ``TableCell`` 相关 API 或模板行生成逻辑。
   - 页眉、页脚、页码异常：优先检查 section header/footer 引用与保存逻辑。
   - 图片位置、缩放、环绕异常：优先检查图片插入策略和当前 API 的能力边界。

3. 修改生成逻辑或模板输入，而不是直接手工改 Word 文档。
4. 生成新的候选文档，例如 ``repair/fix-01/candidate.docx``。
5. 对该候选文档重新执行完整视觉检查，并把证据写回 ``repair/fix-01/evidence/`` 和 ``repair/fix-01/report/``。
6. 若仍不通过，则进入 ``fix-02``；若通过，则把这一轮标记为当前最优候选。

不建议的做法：

- 发现问题后直接在原文档上人工改完就算完成；
- 只修 XML 结构，不回到 Word 验证真实显示；
- 只复查“怀疑页”，不做完整回归；
- 修复时覆盖上一轮唯一证据，导致无法追溯问题来源。


推荐提示词
----------

下面这份提示词适合“只检查，不修复”的场景：

.. code-block:: text

    我要把人工肉眼检查 Word 文档的流程自动化。请严格按下面顺序执行，
    优先使用 windows-mcp / 相关 MCP 工具，不要跳步，不要脑补成功。

    任务目标：
    检查生成后的 Word 文档在最终肉眼可见效果上是否正确，重点看视觉排版、
    元素位置、分页和可见异常；不能只看单元测试、XML、结构树或程序返回值。

    文档路径：
    <文档绝对路径>

    执行要求总原则：
    1. 任何关键动作后都必须做结果验证，验证失败不能继续下一步。
    2. 必须假设用户会同时操作电脑并抢焦点，所以不能默认当前前台窗口就是目标窗口。
    3. 必须假设 Word 可能已启动但没打开目标文档，也可能打开了错误文档。
    4. 每次截图前都必须再次确认当前焦点窗口和当前文档身份。
    5. 不要修改文档，除非我明确要求修复。
    6. 优先保留截图依据；结论必须以截图对应的最终显示效果为准。

    执行顺序：
    - 先验证目标文档路径存在，并输出将要打开的绝对路径。
    - 启动 Word，并显式打开目标文档。
    - 验证 Word 窗口存在、焦点窗口是 Word、当前活动文档就是目标文档。
    - 切到适合检查版式的视图和缩放比例，确保屏幕上真的能看到页面主体。
    - 对每一页或关键页面执行“校验焦点 -> 校验文档 -> Screenshot -> 必要时 Snapshot”。
    - 重点检查标题、正文、表格、单元格纵排/旋转文本、窄格混排、图片、页眉页脚、页码、重叠和分页问题。
    - 如有可疑点，补抓更聚焦截图，并区分真实版式问题和自动化误抓。

    输出格式：
    A. 总结：通过 / 不通过 / 无法判定
    B. 问题列表：
    - 页码：
    - 元素类型：
    - 问题描述：
    - 严重程度：
    - 截图依据：
    - 置信度：
    C. 建议修复方式
    D. 若无问题，也要明确说明“未发现明显版式问题”

下面这份提示词适合“检查失败后继续修复并回归”的场景：

.. code-block:: text

    请先对 Word 文档做严格的视觉排版检查；如果检查不通过，再继续修复生成逻辑，
    并重新生成候选文档做完整回归，直到通过或明确阻塞。不要手工直接修改原始 Word 文档。

    文档路径：
    <文档绝对路径>

    修复约束：
    1. 先完成一轮带截图依据的视觉检查，再决定修复。
    2. 原始文档不得覆盖；每轮修复都输出到 repair/fix-XX/candidate.docx。
    3. 修复依据必须来自上一轮问题报告与截图，而不是猜测。
    4. 每一轮修复完成后，必须重新执行完整视觉检查，不能只看单个问题页。
    5. 如果无法确认当前截图属于目标文档，则不能宣称修复完成。
    6. 对表格问题要特别区分：普通边框/分页问题，还是单元格纵排文本、窄格 RTL/LTR/CJK 混排引起的真实排版异常。

    输出要求：
    - 先给出本轮检查结论；
    - 若不通过，说明问题对应的可能代码层级；
    - 给出本轮修复动作；
    - 给出修复后候选文档路径；
    - 给出回归检查结论；
    - 若仍失败，说明剩余阻塞点和下一轮建议。


结果落地建议
------------

建议把最终结论固定写成一份人工易读的报告，例如 ``report/final_review.md``，
至少包含以下字段：

- 检查对象的绝对路径
- 检查时间
- 证据目录
- 结论：通过 / 不通过 / 无法判定
- 问题列表
- 是否进入修复闭环
- 当前最佳候选文档路径

这样后续再做自动化串联时，检查阶段和修复阶段之间就有稳定的交接面。
如果只是想快速落地，也可以直接参考仓库中的
``docs/automation/final_review_template.md`` 和
``docs/automation/review_result_template.json``。
