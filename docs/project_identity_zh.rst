项目定位（中文）
================

这份文档用于明确 ``FeatherDoc`` 当前 fork 的项目定位、目标边界与演进方向。

一句话概括：

``FeatherDoc`` **不是对 DuckX 的被动镜像维护**，
而是一个以现代 C++、清晰 API、MSVC 可用性和可诊断性为优先目标的
独立演进分支。


当前项目与上游的关系
--------------------

当前仓库最早来源于 ``DuckX``，但现在的目标已经不是“尽量保持历史形态”，
而是：

1. 保留可复用的 ``.docx`` 读写核心能力。
2. 清理旧工程壳子与历史兼容写法。
3. 用更现代的 C++ 风格重新定义公开 API 和实现边界。
4. 围绕当前维护者的实际需求继续演进，而不是回到上游旧设计。

因此，``FeatherDoc`` 应被理解为：

- **代码血缘上**：源自上游导入快照
- **工程方向上**：已经是独立维护的 fork
- **对外品牌上**：以 ``FeatherDoc`` 为准


项目优先目标
------------

当前版本优先做的事情是：

1. 现代化 CMake 与安装导出规则。
2. 使用较新的 C++ 标准和标准库设施。
3. 让 MSVC 构建、运行与测试稳定可复现。
4. 提升 ``open()`` / ``save()`` 等核心路径的错误诊断能力。
5. 优化文档读写过程中的不必要拷贝和大块内存分配。
6. 保持 API 行为明确，减少历史接口语义歧义。


当前明确不优先的事项
--------------------

至少在当前阶段，以下内容不是首要方向：

1. 为了“看起来完整”而堆砌外围工程设施。
2. 为旧 API 长期保留低价值兼容层。
3. 花大量时间维护对旧 CI 平台或历史脚本的形式兼容。
4. 把仓库重新包装成“原始 DuckX 风格”的展示页。


API 方向
--------

``FeatherDoc`` 当前 API 方向强调：

1. 显式状态，而不是隐式副作用。
2. 明确的错误返回，而不是只给 ``bool``。
3. 路径、错误码、格式标志等使用现代标准库类型。
4. 保存行为与另存为行为语义分离。
5. 允许从空文档开始创建，而不是强依赖现成模板文档。


工程方向
--------

当前工程方向强调：

1. Windows / MSVC 可用性必须真实验证，不只是“理论支持”。
2. 测试与文档要跟随 API 一起演进。
3. 第三方依赖版本应保持相对新且可维护。
4. 仓库中的法律说明、赞助入口、贡献流程和项目品牌必须与当前 fork 一致。


建议的对外描述
--------------

对外介绍时，建议使用下面这类表述：

``FeatherDoc`` 是一个面向现代 C++ 的 ``.docx`` 读写库 fork，
重点关注清晰 API、MSVC 可构建性、错误诊断能力和核心路径性能。

如果需要强调血缘关系，可以补充一句：

当前仓库最早来源于 ``DuckX`` 的导入快照，但现在已经按
``FeatherDoc`` 的目标独立演进。
