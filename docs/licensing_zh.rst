许可说明（中文）
================

这份说明文档用于帮助阅读当前仓库的许可结构。

**最终以根目录下的** ``LICENSE`` **和** ``LICENSE.upstream-mit``
**为准**。本文仅做中文解释与使用指引，不替代正式许可文本。

仓库根目录下的 ``NOTICE`` 与 ``LEGAL.md`` 则分别提供“分发摘要”和
“仓库阅读入口”。


当前仓库的许可结构
------------------

当前仓库不是“纯 MIT 开源项目”，而是**分层许可**：

1. ``FeatherDoc`` fork 新增的修改与新增文件

   适用根目录的 ``LICENSE``，属于**非商用、源码可见**条款。

2. 从上游 ``DuckX`` 继承而来的原始代码部分

   原始 MIT 文本保留在 ``LICENSE.upstream-mit`` 中。

3. vendored 第三方依赖

   继续遵循各自原始许可，不因为本仓库的 fork 许可而自动改变。


可以做什么
----------

在不违反 ``LICENSE`` 的前提下，通常可以：

1. 个人学习、阅读、研究源码。
2. 非商用地修改源码并自行使用。
3. 非商用地分发源码或修改版源码，但需要保留许可与归属说明。


不能做什么
----------

未经著作权人书面许可，当前 fork 新增部分**不允许**：

1. 商业使用。
2. 售卖本仓库源码。
3. 售卖修改后的源码。
4. 将其集成到收费软件、收费服务、SaaS、商业 SDK、OEM 或面向客户的商业交付中。
5. 删除、隐藏或篡改版权、署名和许可说明。


如何理解“商用”
--------------

按当前 ``LICENSE`` 的口径，以下场景通常应视为商用：

1. 以盈利、收费、商业优势为目的的使用。
2. 企业内部生产环境部署。
3. 面向客户的项目交付、集成、定制开发或付费支持。
4. 作为付费产品、付费平台或商业服务的一部分对外提供。


如果想商用怎么办
----------------

如果你希望将当前 fork 的新增修改用于商用，需要先取得著作权人
**单独的书面授权**。


关于上游与第三方
----------------

需要特别注意：

1. 你不能通过修改本仓库的 ``LICENSE``，去撤回上游已存在代码的 MIT 授权事实。
2. 你也不能用当前 fork 的许可，覆盖 vendored 第三方依赖原本各自的许可要求。
3. 因此实际分发时，必须同时保留：

   - ``LICENSE``
   - ``LICENSE.upstream-mit``
   - 其他第三方组件要求保留的许可文件或声明


建议的对外表述
--------------

为了避免误导，建议对外使用下面这种描述：

``FeatherDoc`` 当前 fork 的新增修改采用**非商用源码可见许可**；
上游 ``DuckX`` 继承部分保留 MIT 原文说明；第三方依赖遵循各自原始许可。
